\section{Purpose of This Document}

This report explains how the Postgraduate Lifecycle Management System is built and how the complete academic workflow runs from application submission to graduation and record archival. It is written for an administrator or system owner who needs to understand:

\begin{itemize}
  \item what each role does in the system;
  \item what data is created or updated at each phase;
  \item who receives email and in-app notifications;
  \item how the frontend, backend, authentication, database, storage, CSV exports, and reports work together;
  \item what the administrator should know when operating, demonstrating, or maintaining the project.
\end{itemize}

\boxnote{Important Operating Principle}{
The system separates authentication from academic data. Firebase handles login and role claims. Supabase Postgres stores the academic lifecycle records through Prisma. Supabase Storage stores uploaded PDFs using signed URLs.
}

\section{High-Level Project Summary}

The project is a role-based web application for postgraduate lifecycle management. It supports four major user groups:

\begin{itemize}
  \item \role{Applicant}: submits the public application form before having an account.
  \item \role{Student}: submits proposals, ethics approvals, progress reports, theses, and corrections.
  \item \role{Supervisor}: reviews proposals, signs off progress reports, and participates in review panels.
  \item \role{Examiner}: receives thesis assignments, downloads thesis documents securely, and records viva outcomes.
  \item \role{Administrator}: manages admission, users, supervisor/examiner assignments, ethics review, viva scheduling, final thesis archiving, graduation, reports, notification logs, and record archival.
\end{itemize}

The application follows this broad lifecycle:

\begin{center}

\begin{minipage}{0.94\linewidth}

\section{Authentication and Access Control}

\subsection{Login Flow}

\begin{enumerate}
  \item User enters email and password in the login screen.
  \item The frontend signs in using Firebase Authentication.
  \item Firebase returns an ID token with a role claim.
  \item The frontend sends the ID token to \file{/api/auth/session}.
  \item The backend verifies the token with Firebase Admin.
  \item The backend checks the matching Prisma \code{User} record and confirms that the account is active.
  \item A secure HTTP-only session cookie is created.
  \item The user is redirected to the dashboard for their role.
\end{enumerate}

\newpage

\subsection{Role Protection}

Backend routes use a shared authentication wrapper. The wrapper verifies either a bearer token or a session cookie, resolves the current user, checks the allowed roles, and gives the route an authenticated user context.

Examples:

\begin{itemize}
  \item Student-only routes: proposal submission, ethics submission, progress report submission, thesis submission, correction upload.
  \item Supervisor-only routes: proposal evaluation, progress report sign-off.
  \item Examiner-only routes: assigned viva workspace and viva outcome recording.
  \item Administrator-only routes: admission, user management, ethics decision, examiner assignment, viva scheduling, final archive, reports, notification log.
\end{itemize}

\subsection{Document Storage}

PDF files are stored in Supabase Storage. The database stores metadata in the \code{Document} table, including file name, storage path, document type, version, owner links, and soft-delete status.

Protected storage roots:

\begin{itemize}
  \item \code{applications}
  \item \code{proposals}
  \item \code{ethics-approvals}
  \item \code{theses}
  \item \code{progress-reports}
  \item \code{corrections}
\end{itemize}

The application generates signed upload URLs and signed download URLs. This means files are not publicly exposed by default. Download links expire after a short time.

\section{Security and Data Integrity}

\begin{itemize}
  \item \textbf{Role-based authorization}: Routes are restricted to allowed roles before service logic runs.
  \item \textbf{Firebase role claims}: User role is stored in Firebase custom claims and mirrored in Prisma.
  \item \textbf{Active-account check}: Inactive user accounts cannot create sessions.
  \item \textbf{HTTP-only cookies}: Server sessions use secure HTTP-only cookies.
  \item \textbf{Signed storage URLs}: Uploaded and downloaded PDFs use temporary signed URLs.
  \item \textbf{File validation}: Uploads enforce PDF content type and size limits.
  \item \textbf{Soft archival}: Operational records can be archived instead of physically deleted from lifecycle history.
  \item \textbf{Notification audit}: Email delivery attempts are recorded in \code{NotificationLog}.
  \item \textbf{Workflow status validation}: Application and thesis transitions are checked before updates are accepted.
\end{itemize}

\section{Known Implementation Notes}

\begin{itemize}
  \item Applicant accounts are not created at initial application submission. They are created only when the administrator admits the application.
  \item Supabase is used for both database and file storage. Firebase is used for authentication.
  \item The admin dashboard is the visual reference layout for other dashboards.
  \item Some legacy notification paths are email-only, while newer workflow paths use the central notification service for both email and in-app notifications.
  \item Record archival hides records from active operational views but preserves historical data for reporting and audit.
  \item The project has automated tests for core workflows, routes, services, and UI panels.
\end{itemize}

The project implements a complete postgraduate lifecycle:

\begin{enumerate}
  \item Applicant submits application and PDF.
  \item Administrator admits applicant.
  \item System creates Firebase user, Prisma user, student profile, and registration.
  \item Student submits research proposal.
  \item Administrator assigns supervisor.
  \item Supervisor evaluates proposal.
  \item Administrator approves proposal.
  \item Student submits ethics approval.
  \item Administrator approves ethics.
  \item Student submits progress reports.
  \item Supervisor signs off progress.
  \item Student submits thesis.
  \item Administrator assigns examiner and starts examination.
  \item Administrator schedules viva.
  \item Examiner records viva outcome.
  \item Student submits corrections if required.
  \item Administrator approves corrections if required.
  \item Administrator archives thesis and graduates student.
  \item Administrator archives the complete student record when it should leave active workflow lists.
\end{enumerate}

\boxnote{Final Takeaway}{
The system is not only a set of forms. It is a controlled lifecycle engine: each phase changes database state, stores documents securely, validates role permissions, sends notifications, logs delivery attempts, and updates the progress view seen by students and administrators.
}
